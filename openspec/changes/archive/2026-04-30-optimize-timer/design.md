## Context

当前定时器体系已经迁移到 SQLite（见 `timer-db-storage` 规格），但 `timer add` 的入参语义仍偏“开发者友好”而非“使用者友好”：必须提供 `<id>`，且时间字段只能走固定时间格式解析。实际运营/管理场景下，用户更常见的诉求是“先加一个定时器，稍后再通过列表/回执去管理它”，以及“临时延迟 N 秒执行一次”。

本变更需要在不破坏已有时间格式解析的前提下，扩展 `timer add` 的解析逻辑，并确保最终入库的定时器数据可追踪（尤其是自动生成的 ID）。

## Goals / Non-Goals

**Goals:**
- `timer add` 支持省略 `<id>`，由系统生成随机 ID，并在回执中返回。
- `timer add` 支持 `<时间>` 为纯数字时，解释为延迟 \(N\) 秒执行一次（next_trigger = now + N seconds）。
- 自动生成 ID 在数据库唯一约束下可稳定落库（冲突重试），且 `timer list` / `timer remove` 的使用体验不变。

**Non-Goals:**
- 不改变既有的“固定时间/日期时间/重复规则”等解析语义（保持兼容）。
- 不引入新的外部依赖（如雪花算法库等）；优先使用标准库生成随机 ID。
- 不重构整套 TimerManager 调度机制，仅聚焦创建时的输入与持久化。

## Decisions

- **命令语法兼容策略**
  - 决策：保持原 `timer add <id> <时间> <消息>` 语法仍可用，同时新增 `timer add <时间> <消息>` 变体。
  - 依据：避免破坏现有脚本/习惯；对用户来说只需少输入一个字段。
  - 备选：新增单独命令（如 `timer add-auto ...`）——会增加心智负担，放弃。

- **“纯数字时间”的判定**
  - 决策：当 `<时间>` token 满足 `^[0-9]+$` 且转为整数 \(N\) 后 \(N > 0\) 时，解释为延迟秒数。
  - 依据：最大化可预期性，避免把诸如 `20260430` 误判为日期（但这类输入本就不属于既有支持格式时，纯数字更合理）。
  - 备选：支持 `10m/2h` 等单位——需求未提出，留作后续扩展。

- **ID 生成策略**
  - 决策：使用标准库生成短随机 ID（例如 `secrets.token_urlsafe`/`uuid4` 的简化表示），并在入库时处理唯一冲突：冲突则重新生成并重试有限次。
  - 依据：避免可预测 ID；无需引入依赖；与 SQLite 唯一约束天然配合。
  - 备选：基于时间戳的递增 ID——更易冲突/可预测，放弃。

- **回执与可追踪性**
  - 决策：无论是否用户提供 ID，都在成功响应中回显最终 ID；`timer list` 展示 ID 作为主要键；`timer remove` 仍以 ID 为唯一入口。
  - 依据：保证用户“加完就能管”的闭环。

## Risks / Trade-offs

- **[误判风险]** 用户输入的纯数字原本想表达某种日期/时间 → **Mitigation**：仅在既有解析失败或明确命中纯数字规则时走“延迟秒数”；并在回执中明确展示解析后的触发时间，便于发现。
- **[冲突风险]** 随机 ID 与已存在记录冲突 → **Mitigation**：DB 唯一约束 + 冲突捕获重试（次数有限），最终失败返回明确错误。
- **[行为差异]** `timer add` 参数位置变得更灵活，可能影响现有解析边界 → **Mitigation**：以 token 数量/形态（数字 vs 非数字）进行确定性分支，并补充规格场景覆盖。

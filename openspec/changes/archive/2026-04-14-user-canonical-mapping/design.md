## Context

当前系统以 UI 读取到的 `username` 作为数据库用户身份的唯一来源。`UserDAO.get_or_create(username)` 会在 `users` 表中按 `username` 查找或创建用户，因此同一真实用户一旦改名，就会被识别为一个新的 `users.id`。

受影响的业务并不直接依赖 `username`，而是依赖 `users.id` 或 `User.level`：

- `EnterDao` / `ExitDao` / `ReturnDao` 将事件绑定到 `EnterEvent.user_id` / `ExitEvent.user_id` / `ReturnEvent.user_id`
- `KeywordManager` / `KeywordDAO` 将关键字创建者绑定到 `Keyword.creator_id`
- `SeatReservationDAO` 将占座绑定到 `SeatReservation.user_id`
- `CommandManager` 使用 `UserDAO.get_or_create(message_info.nickname)` 获取用户等级做权限判断

因此问题不是“用户名变了”，而是“用户名变化导致稳定身份丢失”。

本次变更选择最小化方案：在 `users` 表中增加一个自关联字段，把“别名用户”映射到“原始用户”，并在统一用户解析入口透明归一化。该方案不尝试自动识别同一人，也不处理历史数据合并，仅在管理员明确建立映射后生效。

## Goals / Non-Goals

**Goals:**

- 为 `users` 表增加“别名用户 → 原始用户”的自关联字段。
- 让依赖 `UserDAO.get_or_create(username)` 的业务在拿到别名用户时，自动改为使用原始用户。
- 保持进入/退出/回归事件、关键字、占座、权限等级等基于 `users.id` 的配置在用户改名后仍然有效。
- 提供一个管理员命令，用于显式把某个用户名绑定为另一个用户名的别名。

**Non-Goals:**

- 不自动推断两个不同 `username` 是否属于同一真实用户。
- 不合并别名用户已有的历史数据；一旦建立映射，后续业务统一按原始用户处理即可。
- 不支持复杂的多级链式映射治理、自动回收孤儿数据、或图状身份合并。
- 不改变现有事件表、关键字表、占座表的外键结构。

## Decisions

### 1. 在 `users` 表增加可空自关联字段 `canonical_user`

选择在 `User` 模型上增加一个可空的自关联外键字段，例如 `canonical_user`。为空表示该记录本身就是原始用户；非空表示该记录是别名用户，其真实身份应归一化到 `canonical_user_id` 指向的用户。

选择这个方案而不是新增 `user_aliases` 表，是因为当前需求只需要“一个用户名可指向一个原始用户”的简单映射，不需要保存复杂别名历史。把映射直接放在 `users` 表上可以减少表数量和查询入口，也更容易让 `UserDAO` 在一次读取后完成归一化。

约束：

- `canonical_user_id` 只能指向一个“原始用户”记录。
- 原始用户的 `canonical_user_id` 必须为空。
- 不允许把用户指向自己。
- 实现上应尽量避免形成链式映射；管理员命令写入前应把目标用户解析为最终原始用户。

### 2. 统一在 `UserDAO.get_or_create(username)` 做透明归一化

当前受影响系统几乎都通过 `UserDAO.get_or_create(username)` 进入数据库身份系统，因此应把 canonical 解析放在这里，而不是分散到关键字、事件、占座、权限等各模块分别处理。

目标行为：

1. 先按 `username` 查找或创建用户记录。
2. 若该用户没有 `canonical_user_id`，直接返回该用户。
3. 若该用户有 `canonical_user_id`，继续读取其原始用户并返回原始用户对象。

这样可以保证以下模块无需了解“别名”概念，也能自然获得稳定用户 ID：

- `CommandManager`
- `KeywordManager`
- `EnterDao` / `ExitDao` / `ReturnDao`
- `SeatReservationDAO` 相关调用链
- 其他调用 `UserDAO.get_or_create` 的位置

选择该方案而不是新增独立的 `resolve_user()` 方法，是为了降低漏改风险。只要沿用现有 `get_or_create` 入口，归一化就天然生效。

### 3. 管理员通过显式命令建立映射

系统无法从 UI 判断两个不同名字是否属于同一真实用户，因此映射必须由管理员手动建立。

建议命令语义：

```text
:alias <别名用户名> <原始用户名>
```

行为：

1. 查找或创建别名用户记录。
2. 查找或创建原始用户记录。
3. 将原始用户名解析到最终原始用户，避免写入链条。
4. 校验别名用户与原始用户不是同一条记录。
5. 将别名用户的 `canonical_user_id` 设为原始用户的 `id`。

不提供自动迁移别名用户既有事件、关键字、占座等数据。根据本次范围约定，只要求映射建立后的后续行为都按原始用户处理。

选择新增单独命令，而不是复用 `admin`/`keyword`/`info` 等既有命令，是因为身份映射是独立的管理动作，语义应当明确且可审计。

### 4. 历史数据保持原状，不做回填迁移

已有的 `EnterEvent`、`Keyword`、`SeatReservation` 等历史记录保持原有 `user_id` 不变。本次变更只保证：

- 原始用户已有配置在其别名触发时继续有效。
- 新建配置时若触发者是别名用户，实际会落到原始用户 ID 上。

不执行“把别名用户旧数据全部改绑到原始用户”的数据库迁移，原因是：

- 用户已明确表示不需要考虑合并。
- 自动迁移存在错误覆盖与歧义风险。
- 透明归一化已足以解决当前最核心的配置失效问题。

### 5. 占座纳入归一化范围，但不修改占座模型

`SeatReservation` 已通过 `UserDAO.get_or_create(username)` 获取用户对象，因此只要归一化在 DAO 入口生效，占座逻辑就能自动跟随原始用户 ID，无需额外改表或改查询接口。

这意味着：

- 别名用户发起占座时，创建/删除/查询都应作用于原始用户。
- 原始用户之前的占座记录对别名用户名继续有效。

## Risks / Trade-offs

- **[管理员映射错误]** → 将导致某个用户名被错误归一化到别的用户。缓解方式是让命令语义足够明确，并在返回消息中展示“别名 → 原始用户”的结果。
- **[历史别名数据仍然挂在旧 ID]** → 建立映射前已创建的数据不会自动迁移。缓解方式是接受该限制，并把本次目标限定为“后续读取和写入统一按原始用户处理”。
- **[链式映射或自引用]** → 可能导致解析复杂化甚至死循环。缓解方式是在命令层禁止自引用，并把目标用户先解析到最终原始用户后再写入。
- **[漏掉未经过 `UserDAO.get_or_create` 的路径]** → 可能仍有个别逻辑绕过归一化。缓解方式是在实现前全局搜索 `User.get_or_create` / `User.get_or_none(username=...)` / `UserDAO.get_by_username` 的使用位置，确认关键路径都在统一入口。
- **[权限等级来源变化]** → 别名用户将继承原始用户的 `level`。这是预期行为，但意味着别名自身记录上的 `level` 将不再参与判断。

## Migration Plan

1. 为 `users` 表新增可空的自关联外键列。
2. 更新 `User` 模型并确保 ORM 初始化时包含该字段。
3. 修改 `UserDAO.get_or_create(username)` 的返回逻辑，使其透明归一化到原始用户。
4. 新增管理员别名命令及其配置。
5. 使用脚本或手工验证以下场景：
   - 原始用户有进入事件，别名用户名进入时仍能触发。
   - 原始用户已有关键字，别名用户名可继续执行。
   - 原始用户已有占座，别名用户名删除/查询占座时命中同一条记录。
   - 别名用户新增关键字/占座时，记录落在原始用户 ID 上。

回滚策略：

- 若仅代码问题，可回滚代码并停止使用别名命令。
- 若已写入 `canonical_user_id`，回滚时需要将对应列置空或保留该列但不再使用。

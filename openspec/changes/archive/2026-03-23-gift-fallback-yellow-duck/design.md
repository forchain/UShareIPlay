## Context

`gift` 命令通过 `UserManager.send_gift()` 实现，流程为：
1. 打开目标用户资料页
2. 点击"送礼物"按钮，进入礼物界面
3. 等待 `give_gift`（赠送）或 `use_item`（使用）按钮出现，以此判断当前是礼物列表还是背包
4. 读取 `luck_item` 获取礼物名称
5. 点击 `give_gift` 或 `use_item` 按钮

当背包为空时，Soul App 会直接展示礼物列表，其中必然包含系统免费礼物"小黄鸭"。与背包礼物不同，礼物列表中的小黄鸭**不会默认选中**，若此时直接点击"赠送"按钮，App 会提示需先选择礼物，界面停留在礼物列表造成流程中断。正确路径是直接点击小黄鸭礼物本身，点击后立即送出并跳回在线列表，无需再点赠送。

## Goals / Non-Goals

**Goals:**
- 在 `send_gift()` 中增加小黄鸭判断分支，识别后走专属"点击即送"路径
- 维持背包有礼物时原有赠送逻辑完全不变

**Non-Goals:**
- 不修改礼物列表的其他选项逻辑
- 不修改命令参数格式或响应模板
- 不为小黄鸭增加单独的配置项（使用现有 `luck_item` 定位即可）

## Decisions

**决策：在 `send_gift()` 内判断而非在 `GiftCommand.process()` 中判断**

礼物名称的读取已经在 `send_gift()` 中完成（`luck_item.text`），在同一处增加分支可最小化改动范围，不需要改动命令层或配置层。

替代方案：在 `config.yaml` 中增加 `yellow_duck_name` 配置 → 不必要的复杂性，"小黄鸭"为系统固定名称，硬编码常量更简单清晰。

**决策：小黄鸭分支中仍调用 `_close_online_drawer()`**

小黄鸭点击后界面会自动跳转，但在线用户抽屉可能仍处于叠加状态，与原流程保持一致，确保后续交互正常。

## Risks / Trade-offs

- [设计说明] 小黄鸭名称固定且始终是礼物列表第一项，无需配置化；名称以模块级常量 `YELLOW_DUCK_NAME = "小黄鸭"` 定义，便于后续维护
- [设计说明] 礼物列表与背包界面的核心区别在于：背包会默认选中第一个礼物（因此直接点击"赠送"可成功），而礼物列表不会自动选中，点击"赠送"会被拦截。检测到 `luck_item` 文本为 `YELLOW_DUCK_NAME` 即可确定当前处于礼物列表界面，直接点击礼物本身绕过赠送按钮

## Migration Plan

- 改动仅限 `send_gift()` 内部，无配置变更，无数据库变更
- 上线即生效，无需迁移步骤
- 回滚：移除小黄鸭分支的 8 行代码即可恢复原状

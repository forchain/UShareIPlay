## Context

本次变更聚焦三类跨模块稳定性问题：

1) **Async 事件循环被阻塞**：运行时主循环与事件兜底逻辑在 asyncio 上下文中运行，但部分路径使用了 `time.sleep(...)` 等阻塞调用。一旦触发，会冻结整个 loop，使 timer、消息队列 drain、事件处理等协程无法推进。

2) **命令解析污染配置**：`CommandParser.parse_command()` 当前会把解析得到的 `parameters` 写回到命令配置 dict（即配置数组中的同一个对象）。这会让后续消息看到被污染的参数，尤其在多来源注入（控制台、Agent 注入、队列、定时器）同时存在时，产生“参数串台/偶现”的高风险缺陷。

3) **driver 恢复后的自动重试边界不清**：`with_driver_recovery` 在捕获 driver 异常后会重建 driver 并立刻重试一次原函数。对于“写操作”（点击/滑动/发送）这可能造成重复副作用；对于“读操作”（获取 page_source / 判断存在）则更可能是期望行为。

现状中已经存在一些稳定性基础设施：
- `AppController.ui_lock` 用于避免后台兜底与命令 UI 流程并发互相干扰。
- `Observability`（events/status）用于写入结构化事件与状态快照，可用于后续验证与回归定位。
- 现有 specs 中 `command-input-normalization` 定义了跨入口的一致规范化规则，但尚未覆盖“解析器不得污染配置对象”的要求。

## Goals / Non-Goals

**Goals:**
- 确保 asyncio 事件循环中不再出现长时间阻塞等待；未知页面退避/节流仍保留但必须非阻塞。
- 命令解析返回值不再修改命令配置对象，保证每条消息解析结果隔离、可并发安全。
- 明确 driver 恢复后的自动重试策略：读操作可自动重试一次；写操作默认不自动重试（只恢复并失败返回/抛错），由上层逻辑决定是否重试。
- 增强可观测性：对“恢复发生/自动重试发生/跳过重试”的决策有清晰日志或事件（复用现有 Observability）。

**Non-Goals:**
- 不重构命令系统/事件系统的整体架构（仍保持动态加载与现有 manager/handler 边界）。
- 不引入新的第三方依赖（例如专门的重试库或 migration 框架）。
- 不对所有 UI 动作做幂等封装（本次仅收敛“自动重试”的默认策略与可控开关）。

## Decisions

1) **Async 非阻塞退避策略**

- 选择：将 asyncio 上下文中的 `time.sleep(...)` 全部替换为 `await asyncio.sleep(...)`。
- 细节：对“连续未知页面”退避从“阻塞 10 秒”改为“在事件循环中 await 退避”，并确保退避不会阻塞其它协程（timer、queue、command dispatch）。
- 备选：
  - 把未知页面退避移到单独线程（复杂且引入更多同步问题）——不选。
  - 完全移除退避（可能造成高频 back/切换抖动）——不选。

2) **命令解析结果返回副本，禁止污染配置对象**

- 选择：`parse_command()` 返回一个新的 dict（或轻量数据结构）用于本次执行，包含 `parameters`，而不是把 `parameters` 写回原配置对象。
- 理由：命令配置是“静态规则”，解析结果是“动态输入”。两者混在同一个可变对象上，会产生跨请求污染。
- 备选：
  - 深拷贝整个命令配置对象（成本略高且不必要）——不选。
  - 用不可变结构（dataclass/frozen）全面替代 dict（收益大但改动面大）——不选（留作后续演进）。

3) **driver 恢复后的重试策略分层**

- 选择：将 driver 恢复装饰器的“是否重试”变成显式策略：
  - 默认：**读操作**装饰器允许“恢复后重试一次”
  - 默认：**写操作**装饰器只恢复，不自动重试
- 具体实现方向（不限定唯一写法）：
  - 增加装饰器参数（例如 `@with_driver_recovery(retry="read")` / `retry=False`）
  - 或拆成两个装饰器：`with_driver_recovery_read` 与 `with_driver_recovery_write`
- 备选：
  - 继续统一自动重试（当前行为）——不选（风险太高）。
  - 完全禁止自动重试（读操作恢复后也不重试）——不选（会降低稳定性与吞吐）。

4) **可观测性策略**

- 选择：复用 `Observability.emit(...)` 记录关键决策点（driver.reinit.* 已存在，补充 “recovery.retry” / “recovery.no_retry” 等），并在关键路径保留 logger 输出以便无 artifacts 环境排查。
- 理由：这些问题多为偶现/现场问题，必须能从 artifacts/日志还原当时决策链。

## Risks / Trade-offs

- **[行为变化] 写操作不再自动重试** → **Mitigation**：在任务中列出受影响的关键写操作入口（点击/发送/滑动），在恢复场景下保证上层能得到明确失败信号；必要时对少数“安全可重试”的写操作显式开启重试。
- **[覆盖不足] 某些阻塞调用不在 EventManager** → **Mitigation**：在实现前做一次搜索/审计，列出所有在 async 路径触发的阻塞点（含 `time.sleep`、潜在长 IO）。
- **[回归风险] 命令解析返回副本可能影响下游依赖 `matching_cmd` 可变字段的代码** → **Mitigation**：将 `command_info` 的消费点作为回归清单，确保使用的是本次解析结果而非全局配置；必要时在 `CommandManager` 层统一封装解析结果结构。
- **[复杂度] 装饰器策略分层需要对现有调用点做分类** → **Mitigation**：先从核心公共方法（page_source / 查找元素 / 点击/输入）分批迁移；用明确命名减少误用。
